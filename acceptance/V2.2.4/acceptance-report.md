### 4b. 前端限制根需求创建任务 — 🐛 BUG（UI 层通过，接口层未拦截）

- **UI 层** ✅：
  - 非项目侧：根需求『2026日常管理』(#14823) 在需求列表中置灰不可点（class 含 `is-disabled`），无法进入其任务页；对照子需求『日常管理\_七月』(#22092) 正常可点、可建任务
  - 排期表侧：多选编辑 → 创建任务 → 抽屉『快速创建任务』→ 选需求弹窗中，根需求均带「请选择其下子节点」提示且不可选中；子需求可正常选中进入创建
  - 截图：final-04b-根需求置灰不可入.jpg、final-04c-排期表根需求禁选.jpg、final-04d-排期表选子需求后步骤3.jpg

- 🐛 **接口层测试（严重）——后端未拦截，前端限制可被绕过**：
  - **测试方法**：复用登录会话，绕过前端 UI 直接 POST `manage_api/project_not_task/add_task`，pid 传根需求『2026日常管理』(#14823)
  - **结果**：返回 `code:0` 创建成功，任务『V224验收-接口级根需求禁建测试』(#22515) **被真实建到根需求 #14823 正下方**（已在需求树中核实其 pid=14823）
  - **预期 / 实际**：预期后端应拒绝在根需求下建任务（返回错误码）；实际后端放行，与本次需求「避免根需求下误建任务导致数据异常」的目标相悖
  - **影响**：前端限制只是遮挡入口，任何绕过前端的调用（脚本、旧版本客户端、并发接口）仍可污染根需求数据，**原数据异常问题存在复发风险**
  - **严重度**：严重（防线不完整）。建议后端在 `add_task` 增加根需求 pid 校验。
  - **复现**：POST `add_task`，body 中 `pid=14823`（或任意根需求 id）→ 观察是否 code:0 成功

- 🐛 **接口层边界测试（一般 + 轻微）**：

  | 用例 | 请求 | 返回 | 判定 |
  |---|---|---|---|
  | 不存在的父需求 pid=99999999 | add_task | `code:50 "sql: no rows in result set"` | 拦截 ✅，但**报错体把底层 SQL 错误直接抛给前端**（轻微信息泄露，建议改为业务化文案） |
  | 空任务名称 | add_task name="" | `code:51 "非项目需求下的任务名称不能为空"` | 正确拦截 ✅ |
  | 负数计划工时 plan_hour=-5 | add_task | `code:0` 成功，任务『V224边界-负工时(可删除)』(#22516) 落库 `task_hour=0 / plan_hour=-5` | 🐛 **轻微（前端已挡）** — 前端数字步进器限制、正常操作填不了负数，此问题仅在绕过前端直调接口时出现；task_hour 被归零但 plan_hour 负值直接入库，属后端缺兜底校验，建议防御性补 ≥0 校验 |

- 【发版素材】修复部分成员在『2026日常管理』根需求下误建任务导致数据异常的问题，异常数据已转移；排期表与非项目创建任务时前端已限制根需求不可选，只能在子需求下建任务，日常管理需求恢复正常创建。（注：接口层后端校验建议一并补上，见验收 BUG）
