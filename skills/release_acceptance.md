# 51PM — 版本验收（依开发内容走流程验收，产出验收报告 + 截图，供发版技能使用）

> **半自动验收 skill**。输入是开发者的本周开发内容（可能很糙），agent 扮演真实用户在 51PM 里把每个功能流程走一遍，检测 BUG 与内容不合理，边走边截图，最终产出结构化验收报告。报告 + 截图交给发版技能（同目录 [release_notes.md](release_notes.md)）生成发版内容初稿，最后由用户人工定稿。

## 适用场景

- "验收本周开发内容：……"
- "帮我验收 V2.2.9，开发完成了这些：……"
- "测试环境走一遍这几个新功能，看有没有问题"

## ⚠️ 与其他 51pm skill 的关键差异：必须走真实 UI 交互

README 里"首选 Vue data 直写"的规则**在本 skill 中反转**：

| 用途                               | 允许的手段                                                                                                                             |
| ---------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| 操作功能（点击、输入、切换、拖拽） | **只能走真实 DOM 交互**（`click` / `type` / 键盘事件）——验收的目的就是验证 UI 交互本身，Vue 直写会把待验证的交互层整个绕过去，等于没验 |
| 核对数据（列表条数、字段值、状态） | 可以用 `js()` 读 Vue data 作为辅助断言，但不能代替 UI 操作                                                                             |

如果一个按钮用真实点击点不动——**这本身就是一个 🐛BUG 发现**，记录下来，不要换 Vue 直写绕过去。

## 参数

| 参数         | 示例                 | 说明                                                                                                                      |
| ------------ | -------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `env`        | `test` / `prod`      | **验收固定测试环境** `10.67.8.183:7777`（有"当前为开发环境"水印）。用户明确说了才用正式环境；正式环境下所有写操作逐项先问 |
| `version`    | `V2.2.9`             | 本次验收的目标版本号，用于截图目录命名                                                                                    |
| `dev_notes`  | （文本）             | 开发者的本周开发内容，允许口语化、不完整                                                                                  |
| `output_dir` | `acceptance/V2.2.9/` | 截图与报告落盘目录（仓库内相对路径，绝对路径 `d:\project\51PM验收-测试-发版\acceptance\{版本}\`），默认按版本号建         |

## 前置（强制）

1. **浏览器页签**：优先复用已共享的 51pm 页签；开新页签后**第一件事** `page.setViewportSize({width:1920,height:1080})`（默认 1280×720 会裁掉右侧栏、左侧仪表盘菜单不渲染）。
2. **登录态注入**：新页签先开同源静态页（如 `/favicon.ico`，SPA 初始化会 401 跳登录页），注入 `regression/auth/state.json` 里测试环境 origin 的 localStorage（至少 `oauthToken`）再进 app；若被踢到登录页，企微 SSO 通常能自动续登（CAS 回调自动完成），等待即可；**续登失败才暂停问用户扫码**，不代输凭据。
3. **环境固定测试环境**：一律跑 `10.67.8.183:7777`，不再询问；仅用户明确说「正式环境」才切换。后端 8888 需本机转发：`cd regression; npm run proxy` 常驻。
4. **入口先查表**：dev_notes 里每条功能先查 [entry_map.md](entry_map.md)；查不到的在清单里标"待现场找"直接现场探索（不等用户勾选）。用户口语化输入模块归属实在无法推断时才追问——反例：曾因"日报导出"入口猜错浪费大量轮次，优先查表而非满站猜。
5. **权限门禁（仅限造前置数据环节）**：遇前端角色/白名单禁用按钮（如 PM 审批 `systemRole!=='PM'`、QA递交 `testListRooters`），**不要切换角色**（切回可能重新扫码）、不要对禁用按钮死循环重试；读按钮 Vue 组件找背后方法直调打开表单，后续仍真实交互；「前端禁用但后端放行」作为 ⚠️ 写进报告。被验功能本身的权限问题不得绕，记 ⚠️。
6. **已知交互坑（V2.2.6 轮实测）**：①上传触发的 filechooser **必须消费**（`setFiles` 或开新页签甩掉），不消费会一直挂起，导致页面随机跳回 main、点击失灵；②节日主题动画（如圣诞飘雪）会令 `locator.click` 永远报「元素不稳定」，用 `force:true` 或坐标点击；③ `tb_publish` / `myMessageBox` / `my_board` 等子页直接 URL 或 `router.push` 会被重定向回 main，必须真实点左侧菜单（`el-menu-item` 的 vm.handleClick 可作兜底）。

## 用户侧交付约定

0. **提及实体一律给名称**：报告/消息里引用项目、任务、需求、缺陷等实体时，必须写「名称（#ID）」格式，如「任务『日报导出优化』(#12345)」；**只给裸 id 用户看不懂**。名称从页面标题/列表行读取。
1. **产物路径**：报告与截图全部落盘本仓库 `acceptance/{版本}/`，给用户发路径用 `D:\project\51PM验收-测试-发版\acceptance\{版本}\`（Windows 格式）。
2. **截图只留强相关**：只截三类——a) 功能验证的关键结果页（每功能 1~2 张）；b) BUG 现场（必截）；c) 定妆图 final-\*。中间过程（下拉展开、每次输入、探索性打开的页面）**不截**。反例：曾一轮截 60+ 张，远超需要。
3. **时间范围遵守页面提示**：设日期范围后如页面出现「日期范围过大，时间轴已禁用/请选择一个月以内的日期范围」等提示，**立即缩小到提示要求的范围内重查**，不要带着禁用状态继续验（会漏验时间轴等联动组件）。通用原则：任何筛选后先读页面 warning/alert 文案再继续。

## 流程

### 第 1 步：把开发内容拆成验收清单（输出后直接开跑）

对 `dev_notes` 逐条解析，产出验收清单，每项包含：

```
- 功能名：（从开发内容里提炼）
- 推测入口：「一级菜单-二级页面」（参考 README 主模块表；拿不准就标"待现场找"）
- 计划流程：作为真实用户会怎么走（3~8 步）
- 观察点：这条功能"对/不对"分别长什么样
```

**清单输出后直接开始验收，不等用户确认**。仅当开发内容有明显歧义、无法推进时才提问；用户看到清单后若有纠正会随时打断。数据前置条件缺失（测试项目没需求/没任务等）直接补建数据继续，不需询问。

### 第 2 步：逐项走流程 + 截图

每个验收项按"计划流程"用真实 UI 操作走一遍，检查维度：

| 维度       | 看什么                                                                                                                                                                                                                                                              |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 功能正确性 | 按钮/入口存在、点击有响应、流程能走通、无报错弹窗、无控制台红错                                                                                                                                                                                                     |
| 数据正确性 | 数字对得上（可交叉核对：看板汇总 vs 明细列表条数）、状态流转正确                                                                                                                                                                                                    |
| 交互体验   | 加载有反馈、操作路径顺不顺、有没有点了没反应/重复提交                                                                                                                                                                                                               |
| 内容合理性 | 文案有无歧义/错别字、字段口径怪不怪、与相邻功能是否矛盾                                                                                                                                                                                                             |
| 边界情况   | 空数据展示、超长文本溢出、特殊字符、翻页、筛选组合、日期跨月/跨年、0 与极大数值                                                                                                                                                                                     |
| 接口层     | 关键操作时监听页面请求/响应（`page.on('request'/'response')` 或 `performance.getEntriesByType('resource')` 拿真实接口路径）；再用 `page.evaluate` 内 `fetch` 复用会话直调接口验边界参数（非法值、越权 id、空参）；注意前端走 `localhost:8888` 代理，直调用同一 host |
| 数据一致性 | 前端展示 vs 接口响应 vs 跨页面交叉核对（看板汇总 vs 明细条数）；写操作后刷新页面确认落库                                                                                                                                                                            |

> 验收**不只是浏览器上点一遍**：每个功能默认覆盖「UI 流程 + 边界情况 + 接口层 + 数据一致性」四层；哪层验不了（如无接口文档、无权限）在报告里标 ⚠️ 写明原因，不要静默跳过。
> **接口测试的写位规则**：**明细归功能、汇总归独立节**——
> ① 属于某功能的接口验证（路径+关键参数+结果）直接写进该功能节「覆盖层-接口」一行，不另开表；
> ② 独立节固定叫「接口测试汇总与未覆盖声明」，只放三样功能节装不下的：跨功能/附带发现（如接口参数不过滤、前后端权限不同步）、未覆盖范围如实声明（写接口边界/极限值/特殊字符）、api spec 指针；
> ③ **禁止把 UI 验收结果换个说法当接口用例凑表**（如"项目编辑(间接)"这类行，纯重复）。
> 同时：接口路径+参数+坑回填 entry_map 对应行备注，用例（含完整参数）固化进 `api-v{版本}.spec.js`。

**截图规范**（`page.screenshot({path, type:'jpeg', quality:80})` 直接落到 `output_dir` 绝对路径）——**默认精简模式，截图是最大耗时项**：

```
{序号:02d}-{功能名}-bug-{简述}.jpg    # 异常图：发现问题必截，截到问题现场
final-{功能名}.jpg                    # 定妆图：每个功能 1 张，功能最完整的一屏，给发版文档用
{序号:02d}-{功能名}-{步骤描述}.jpg    # 过程图：仅两种情况才截——①写操作提交前后 ②用户明确要求"全程截图"
```

- 核对文本/数字/状态一律用 `page.evaluate` 读 DOM/Vue data 断言，**不要靠截图肉眼比对**；截图只留给"人需要看的证据"（BUG 现场、定妆图）
- **交付截图一律用 headless Playwright 脚本截，禁止用集成浏览器出图**：
  - 根因：集成浏览器视口受 VS Code 面板物理尺寸限制，`setViewportSize(1920)` 后 `innerWidth` 瞬时返回 1920 但会被**异步静默回弹**到 ~1390，截前断言存在假阳性窗口；`page.screenshot` 仍按 1920 画布输出 → 右/下大片留白
  - 做法：集成浏览器只做探索交互与数据断言；验收尾声用 `regression/scripts/headless-login.js` 的 `launchLoggedIn()`+`shot()`（SSO 免扫码登录、公告弹窗、视口断言已封装）写一个只读重放脚本，把全部定妆图/BUG 图一次截完（写操作已完成的场景可接口推回前置状态→截图→重演复原，V2.2.7 reshoot.js 先例）
  - headless 下无坐标偏移，hover/click 均可真实操作；SESSION 过期时脚本会明确报错提示 `npm run login`

**速度纪律**（防止一次验收拖数小时）：

- 复验轮次默认 **1 轮**；只有想复现"偶发"BUG 才多轮往返
- 一次消息建议只带 **1~2 个功能**；功能多了必撞迭代上限被截断，续跑重读上下文更慢
- 数据核对用 `page.evaluate` 一次性批量读值断言，不要逐条截图核对

### 第 3 步：结论分级

每个验收项给一个结论：

- ✅ **通过** — 流程走通、各维度无异常
- 🐛 **BUG** — 必须给出：复现步骤（从哪个页面开始逐步列）、预期 vs 实际、截图、严重度（阻断 / 严重 / 一般 / 轻微）
- ⚠️ **需人工复核** — agent 验不了或拿不准的，写明原因：
  - 权限不够（当前账号看不到该页面/数据）
  - 时间/后端触发逻辑（如"超两周自动转暂停中"只能验展示层）
  - 视觉细节拿不准（截图给用户判断）
  - 不确定是 BUG 还是设计如此

### 第 4 步：产出验收报告

在 `output_dir` 写 `acceptance-report.md`，固定模板：

```markdown
# 51PM {version} 验收报告

- 验收时间：YYYY-MM-DD HH:mm
- 环境：测试 10.67.8.183:7777
- 验收账号角色：XXX
- 覆盖层：每个功能默认覆盖 UI 流程 / 边界 / 接口 / 数据一致性 四层
- 总览：N 项验收，✅x / 🐛y / ⚠️z

## 0. 原始验收需求（开发者提交的本周开发内容）

> {原样引用用户给的 dev_notes 全文，含开头"帮我验收 VX.X.X…"和结尾"要求：…出验收报告"}

对应关系：需求 1 → §1，需求 2 → §2 …（列出每条需求映射到哪一节）

## 1. {功能名} — ✅通过

- 入口：「统计-项目人员看板」
- 实走流程：1)… 2)… 3)…（涉及的项目/任务/需求写「名称（#ID）」）
- 覆盖层：UI ✅ / 边界 ✅（验了什么边界写括号里）/ 接口 ✅（**接口路径+关键参数+结果写这里**，如 `xx/get_list?id=N` 正常+非法id 均无 5xx）/ 数据 ✅（验不了的层标 ⚠️+原因）
- 推断需关注人员：PM、组长

## 2. {功能名} — 🐛BUG

- 复现步骤：…
- 预期：… / 实际：…
- 严重度：严重

## 3. {功能名} — ⚠️需人工复核

- 原因：当前账号无 DTA 管理员权限，看不到模型数据看板明细
- 已验部分：入口存在、页面能打开（截图 03-xxx-入口.jpg）

## 接口测试汇总与未覆盖声明

{只放功能节装不下的三样：跨功能/附带发现表（# | 发现 | 证据 | 建议）、未覆盖范围如实声明、api spec 指针；
各功能自己的接口验证明细已在对应节「覆盖层-接口」，不在此重复}

## 交付前需人工确认（汇总）

{🐛/⚠️ 项汇总表：# | 事项 | 建议；开头「总览」行的 ⚠️ 计数要与本表口径一致（功能级⚠️计入总览，附随发现另说“另有 N 项待确认”，不要两处数字对不上）}

## 验收产生的测试数据

{列出本轮写操作产生的数据：项目#ID + 递交/动态/文档/字段变更……供下轮回归用例作数据前置条件引用}

## 发版内容（初稿）

{第 5 步产出后回填到这里——发版内容不落独立文件，必须写进本报告，这样 md 和 html 是同一份完整交付物；
⚠️ 版本块用 `###`/`####` 真标题分层（如 `### V2.2.6 发布于…` `#### 新增功能`），不要用 `**粗体**` 行冒充标题——否则 html 无层级、目录不可导航}
```

**报告转 HTML**（pandoc 在 WSL，零 token 成本）。⚠️ **执行时机：第 5 步「发版内容」写回报告之后**，不要第 4 步写完 md 就转——否则 html 里缺发版内容节：

```bash
cd {output_dir} && printf '<style>\n  body { max-width: 72em; }\n  table { display: table; width: 100%%; }\n</style>\n' > _report-style.html && pandoc acceptance-report.md -f gfm -t html5 --standalone -H _report-style.html --metadata title="51PM {version} 验收报告" --metadata lang=zh-CN --toc -o acceptance-report.html
```

- Windows 下 pandoc 在 WSL：`wsl -e bash -c "cd /mnt/d/... && pandoc ..."`；md/样式文件若被外部清理，重建样式再转
- `--toc` 必带：报告节多，无目录难导航

- 必须带 `-H _report-style.html`：pandoc 默认模板正文宽度写死 36em，太窄，表格挤成一条
- md 里列表项内嵌表格时，表格前**必须空一行**，否则 gfm 解析会把表格压平成带竖线的段落文字
- 图片是相对路径引用，html 与截图同目录，双击即可带图浏览；对外发送时把整个 `{output_dir}` 目录打包
- 转换失败不阻断流程，md 仍是主产物，报告里提一句即可
- 发版内容后续被人工定稿修改回 md 后，重跑同一条命令重转 html 即可

**回填入口表（强制，报告交付前做，不等用户提醒）**：本轮验收中**新确认的入口**和**纠正的入口**，立即按回填规则写入同目录 [entry_map.md](entry_map.md)（全 51pm skill 共享的入口地图）；同页面踩到的新坑以 ⚠️ 备注写进对应行。每轮验收都执行。

### 第 5 步：产出发版内容（**默认必做**：验收报告写完后直接生成发版内容初稿，不等用户追问；存在 🐛/⚠️ 待确认项时照样产出，正文按"已生效部分"的口径写，并附「交付前需人工确认」表说明哪些措辞待确认后调整）

> ⚠️ **强制：写发版内容时必须真正读取发版 skill（**同目录 [release_notes.md](release_notes.md)**）并逐条套用，不许读完凭感觉写。** 用户会拿真实定稿版比对，分类/强度判错会被当场发现。

**产出位置**：写入 `acceptance-report.md` 的「## 发版内容（初稿，待人工定稿）」节，**不落独立文件、不只在消息里发一遍**；写完后再执行第 4 步的 pandoc 命令转 html——验收报告和发版内容初稿是**同一份 html** 交付。

落笔前必须逐条对照的 4 个 skill 判断点（历轮高频判错处）：

1. **分类：新增功能 vs 体验优化** —— 套用 skill 的「判断表」。**产出物是全新能力/独立模块/独立页面/独立文档位 → 新增功能**，哪怕它挂在已有页面上、动机是"提升体验"也算新增功能。反例教训：`创建人员群组`、`项目测试文档` 是全新能力，应归**新增功能**，我却因"挂在已有页面"错判成体验优化。
2. **影响强度** —— 套用 skill 的「强度规则表」。新增用户可感知的核心能力 → 至少**中等**，不要保守判"较小"。强度按 **dev_notes 全部条目**判（含未标"体现到发版"、未验收或验不了的条目——dev_notes 就是本次发版范围，**不要去翻发版记录/历史版本确认"全集"**，范围存疑就在初稿标 ⚠️ 留待定稿）。统一用 `较小/中等/较大`。
3. **BUG 转正向表述** —— 对外发版**不暴露"曾出数据异常/误建"等负面信息**。把"修复了XX异常"改写成正向能力，如"仅支持在子需求下创建任务，规范需求选择范围，防止错选"。内部 BUG（尤其接口层发现）放到报告的「交付前需人工确认」节，不进对外发版正文。
4. **价值括号写业务结果，不写功能能力** —— 套用 skill 的「价值描述红黑榜」。写"降低XX创建成本/减少人工汇总/提高盘点效率"，不写"方便统一录入/支持XX"（能接在"支持"后的多半是功能能力，不合格）。

写完后**用 skill 的「常见错误」表自查一遍**再交付。发版内容以用户人工定稿为最终版；被纠正后把差异作为反例回填本节。

### 第 6 步：交付

- 报告与截图目录路径发回用户（Windows 格式 `D:\project\51PM验收-测试-发版\acceptance\{版本}\`），不需询问。
- 定妆图不需要另行拷贝：发版内容直接引用 `acceptance/{版本}/final-*.jpg` 原路径（不设暂存目录，验收产物和发版素材同源）。

## 安全红线

1. 验收会产生写操作（建任务、改状态等）——**测试环境**允许直接做，测试数据**不需要清理**（但要记录在报告「验收产生的测试数据」节供回归引用）；**正式环境**每个写操作前单独问用户。
2. 不删除任何已有数据，包括测试环境。
3. 验收中遇到登出：先等企微 SSO 自动续登（CAS 回调通常自动完成）；续登失败/需扫码/二次验证才暂停问用户，不代输凭据。
4. 两套环境 host 不外发（不写进给外部看的文档/截图标注）。

## 测试数据策略

测试环境与正式环境**数据库完全隔离**，测试库里的"真实项目"只是副本——**可随便挑合适的项目直接写数据，无污染顾虑**，不需要专用测试项目。测试库会不定期整体刷新（先例：原测试项目 SJ202607100001 记录消失、#6712 易主），因此：

- **只读用例**：不硬依赖固定 ID 的存在性——验收时动态挑选符合条件的数据（如"扫到任意一条指派给离职者的完工任务即可"）；沉淀回归用例时采用「动态发现 → 写死当时找到的 ID → 失效时（fail 提示）重扫换一个」模式，fail 消息里写清重建/重扫方法。
- **写链路用例**：随便挑一个业务形态合适的项目/对象直接写（如给某真实发包造反馈），写入内容**必须带版本前缀标识**（如「V2.2.7验收-」）便于识别，不清理；用掉的数据记录进报告「验收产生的测试数据」节。
- 数据前置缺失不是异常而是常态，直接补建/重找继续跑，不询问（同总控 SKILL「不问直接干」③）。
