# 51PM — 版本验收（依开发内容走流程验收，产出验收报告 + 截图，供发版技能使用）

> **半自动验收 skill**。输入是开发者的本周开发内容（可能很糙），agent 扮演真实用户在 51PM 里把每个功能流程走一遍，检测 BUG 与内容不合理，边走边截图，最终产出结构化验收报告。报告 + 截图交给发版技能（同目录 [release_notes.md](release_notes.md)）生成发版内容初稿，最后由用户人工定稿。

## 适用场景

- "验收本周开发内容：……"
- "帮我验收 V2.2.9，开发完成了这些：……"
- "测试环境走一遍这几个新功能，看有没有问题"

## ⚠️ 与其他 51pm skill 的关键差异：必须走真实 UI 交互

README 里"首选 Vue data 直写"的规则**在本 skill 中反转**：

| 用途 | 允许的手段 |
|---|---|
| 操作功能（点击、输入、切换、拖拽） | **只能走真实 DOM 交互**（`click` / `type` / 键盘事件）——验收的目的就是验证 UI 交互本身，Vue 直写会把待验证的交互层整个绕过去，等于没验 |
| 核对数据（列表条数、字段值、状态） | 可以用 `js()` 读 Vue data 作为辅助断言，但不能代替 UI 操作 |

如果一个按钮用真实点击点不动——**这本身就是一个 🐛BUG 发现**，记录下来，不要换 Vue 直写绕过去。

## 前置（强制）

1. **CDP 探活 + Tab 复用模板**：开跑第一步先 `list_tabs()` 探活——失败/超时说明 automation Chrome 没开，**立即停下提示用户跑 `ensure-chrome.sh`**，不要带着断连状态走到操作步才报错。探活通过后执行 [README.md → 操作前置规则](README.md#操作前置规则强制)，禁止盲目 `new_tab`。
2. **登录**：走企微 OAuth。遇到登出状态**暂停问用户**，不代输凭据。
3. **环境确认**：用户没写明环境时，`AskUserQuestion` 问清楚（验收一般在测试环境，但要确认这批功能是否已部署到测试环境）。
4. **入口先查表再问**（2026-07-08 教训）：dev_notes 里每条功能先查 [entry_map.md](entry_map.md)；查不到且没写明「模块-页面」入口的，**在第 1 步清单确认时逐条列出推测入口让用户勾选/纠正**，不要开跑后自己满站猜——V2.2.3 验收时"日报导出"入口猜错（统计→工时页 vs 实际在任务→查看日报弹窗），浪费了大量轮次。用户口语化输入时主动追问模块归属。

## 用户侧交付约定（2026-07-08 / 07-09 用户反馈固化）

0. **提及实体一律给名称**（2026-07-09）：报告/消息里引用项目、任务、需求、缺陷等实体时，必须写「名称（#ID）」格式，如「任务『日报导出优化』(#12345)」；**只给裸 id 用户看不懂**。名称从页面标题/列表行读取。
1. **路径必须给 Windows 可访问格式**：验收产物目录（`agent-workspace/acceptance/`）已是指向 D 盘仓库的 symlink，给用户发路径一律用 `D:\project\hermes-agent\AgentGroups\BrowserHarness\agent-workspace\acceptance\{版本}\` 格式；仅当文件确实在 WSL home 时才用 `\\wsl.localhost\Ubuntu\home\dengxinyu\...` 格式，**不要给 `/home/dengxinyu/...` 裸 WSL 路径**。飞书场景优先用 MEDIA: 直接发图（MEDIA 用 WSL 路径没问题，是 agent 本地读取），但报告里引用的目录路径要给 Windows 格式。
2. **截图只留强相关**：只截三类——a) 功能验证的关键结果页（每功能 1~2 张）；b) BUG 现场（必截）；c) 定妆图 final-*。中间过程（下拉展开、每次输入、探索性打开的页面）**不截**。V2.2.3 一轮截了 60+ 张，用户明确说太多了。
3. **时间范围遵守页面提示**：设日期范围后如页面出现「日期范围过大，时间轴已禁用/请选择一个月以内的日期范围」等提示，**立即缩小到提示要求的范围内重查**，不要带着禁用状态继续验（会漏验时间轴等联动组件）。通用原则：任何筛选后先读页面 warning/alert 文案再继续。

## 流程

### 第 1 步：把开发内容拆成验收清单（先确认再动手）

对 `dev_notes` 逐条解析，产出验收清单，每项包含：

```
- 功能名：（从开发内容里提炼）
- 推测入口：「一级菜单-二级页面」（参考 README 主模块表；拿不准就标"待现场找"）
- 计划流程：作为真实用户会怎么走（3~8 步）
- 观察点：这条功能"对/不对"分别长什么样
```

**把清单发给用户确认一次**（飞书场景直接回消息），确认后再开始操作。这一步防止对开发内容理解跑偏，白跑一轮。

### 第 2 步：逐项走流程 + 截图

每个验收项按"计划流程"用真实 UI 操作走一遍，检查维度：

| 维度 | 看什么 |
|---|---|
| 功能正确性 | 按钮/入口存在、点击有响应、流程能走通、无报错弹窗、无控制台红错 |
| 数据正确性 | 数字对得上（可交叉核对：看板汇总 vs 明细列表条数）、状态流转正确 |
| 交互体验 | 加载有反馈、操作路径顺不顺、有没有点了没反应/重复提交 |
| 内容合理性 | 文案有无歧义/错别字、字段口径怪不怪、与相邻功能是否矛盾 |
| 边界情况 | 空数据展示、超长文本溢出、特殊字符、翻页、筛选组合、日期跨月/跨年、0 与极大数值 |
| 接口层 | 关键操作时用 harness 的网络请求能力（见 `interaction-skills/network-requests.md`）看请求/响应：状态码、报错体、慢接口；必要时 `js()` 内 `fetch` 复用会话直调接口验边界参数（非法值、越权 id、空参） |
| 数据一致性 | 前端展示 vs 接口响应 vs 跨页面交叉核对（看板汇总 vs 明细条数）；写操作后刷新页面确认落库 |

> 验收**不只是浏览器上点一遍**：每个功能默认覆盖「UI 流程 + 边界情况 + 接口层 + 数据一致性」四层；哪层验不了（如无接口文档、无权限）在报告里标 ⚠️ 写明原因，不要静默跳过。

**截图规范**（用 harness 的 `screenshot()`，落到 `output_dir`）——**默认精简模式，截图是最大耗时项**：

```
{序号:02d}-{功能名}-bug-{简述}.jpg    # 异常图：发现问题必截，截到问题现场
final-{功能名}.jpg                    # 定妆图：每个功能 1 张，功能最完整的一屏，给发版文档用
{序号:02d}-{功能名}-{步骤描述}.jpg    # 过程图：仅两种情况才截——①写操作提交前后 ②用户明确要求"全程截图"
```

- 核对文本/数字/状态一律用 `js()` 读 DOM/Vue data 断言，**不要靠截图肉眼比对**；截图只留给"人需要看的证据"（BUG 现场、定妆图）

**速度纪律**（防止一次验收拖数小时）：

- 复验轮次默认 **1 轮**；只有想复现"偶发"BUG 才多轮往返
- 一次消息建议只带 **1~2 个功能**；功能多了必撞迭代上限被截断，续跑重读上下文更慢
- 数据核对用 `js()` 一次性批量读值断言，不要逐条截图核对

### 第 3 步：结论分级

每个验收项给一个结论：

- ✅ **通过** — 流程走通、各维度无异常
- 🐛 **BUG** — 必须给出：复现步骤（从哪个页面开始逐步列）、预期 vs 实际、截图、严重度（阻断 / 严重 / 一般 / 轻微）
- ⚠️ **需人工复核** — agent 验不了或拿不准的，写明原因：
  - 权限不够（当前账号看不到该页面/数据）
  - 时间/后端触发逻辑（如"超两周自动转暂停中"只能验展示层）
  - 视觉细节拿不准（截图给用户判断）
  - 不确定是 BUG 还是设计如此

### 第 4 步：产出验收报告

在 `output_dir` 写 `acceptance-report.md`，固定模板：

```markdown
# 51PM {version} 验收报告
- 验收时间：YYYY-MM-DD HH:mm
- 环境：测试 10.67.8.183:7777
- 验收账号角色：XXX
- 覆盖层：每个功能默认覆盖 UI 流程 / 边界 / 接口 / 数据一致性 四层
- 总览：N 项验收，✅x / 🐛y / ⚠️z

## 0. 原始验收需求（开发者提交的本周开发内容）
> {原样引用用户给的 dev_notes 全文，含开头"帮我验收 VX.X.X…"和结尾"要求：…出验收报告"}

对应关系：需求 1 → §1，需求 2 → §2 …（列出每条需求映射到哪一节）

## 1. {功能名} — ✅通过
- 入口：「统计-项目人员看板」
- 实走流程：1)… 2)… 3)…（涉及的项目/任务/需求写「名称（#ID）」）
- 覆盖层：UI ✅ / 边界 ✅ / 接口 ✅ / 数据 ✅（验不了的层标 ⚠️+原因）
- 截图：01-xxx-进入.png、01-xxx-下钻.png、final-xxx.png
- 推断需关注人员：PM、组长
- 【发版素材】支持XXX，支持XXX，方便XXX；   ← 按发版技能的句式预写一版

## 2. {功能名} — 🐛BUG
- 复现步骤：…
- 预期：… / 实际：…
- 截图：02-xxx-bug-数据不一致.png
- 严重度：严重

## 3. {功能名} — ⚠️需人工复核
- 原因：当前账号无 DTA 管理员权限，看不到模型数据看板明细
- 已验部分：入口存在、页面能打开（截图 03-xxx-入口.png）

## 交付前需人工确认（汇总）
{🐛/⚠️ 项汇总表：# | 事项 | 建议}

## 发版内容（初稿，待人工定稿）
{第 5 步产出后回填到这里——发版内容不落独立文件，必须写进本报告（2026-07-09/07-10 用户约定），这样 md 和 html 是同一份完整交付物}

## 交接给发版技能
定妆图清单 + 各功能【发版素材】段 + 上方「发版内容」初稿，可直接作为发版定稿的输入。
```

**报告转 HTML**（pandoc 3.7 已装在 WSL，零 token 成本）。⚠️ **执行时机：第 5 步「发版内容」写回报告之后**，不要第 4 步写完 md 就转——否则 html 里缺发版内容节（2026-07-10 用户反馈的真实翻车）：

```bash
cd {output_dir} && printf '<style>\n  body { max-width: 72em; }\n  table { display: table; width: 100%%; }\n</style>\n' > _report-style.html && pandoc acceptance-report.md -f gfm -t html5 --standalone -H _report-style.html --metadata title="51PM {version} 验收报告" --metadata lang=zh-CN -o acceptance-report.html
```

- 必须带 `-H _report-style.html`：pandoc 默认模板正文宽度写死 36em，太窄，表格挤成一条
- md 里列表项内嵌表格时，表格前**必须空一行**，否则 gfm 解析会把表格压平成带竖线的段落文字
- 图片是相对路径引用，html 与截图同目录，双击即可带图浏览；对外发送时把整个 `{output_dir}` 目录打包
- 转换失败不阻断流程，md 仍是主产物，报告里提一句即可
- 发版内容后续被人工定稿修改回 md 后，重跑同一条命令重转 html 即可

**回填入口表（强制，报告交付前做，不等用户提醒）**：本轮验收中**新确认的入口**和**纠正的入口**，立即按回填规则写入同目录 [entry_map.md](entry_map.md)（全 51pm skill 共享的入口地图）；同页面踩到的新坑以 ⚠️ 备注写进对应行。每轮验收都执行。

### 第 5 步：产出发版内容（**默认必做**，2026-07-10 用户反馈固化：验收报告写完后直接生成发版内容初稿，不要等用户追问；存在 🐛/⚠️ 待确认项时照样产出，正文按"已生效部分"的口径写，并附「交付前需人工确认」表说明哪些措辞待确认后调整）

> ⚠️ **强制（2026-07-09 用户明确要求）：写发版内容时必须真正读取发版 skill（**同目录 [release_notes.md](release_notes.md)**）并逐条套用，不许读完凭感觉写。** 用户会拿真实定稿版比对，分类/强度判错会被当场发现。

**产出位置（2026-07-10 固化）**：写入 `acceptance-report.md` 的「## 发版内容（初稿，待人工定稿）」节，**不落独立文件、不只在消息里发一遍**；写完后再执行第 4 步的 pandoc 命令转 html——验收报告和发版内容初稿是**同一份 html** 交付。

落笔前必须逐条对照的 4 个 skill 判断点（这些正是 2026-07-09 我判错的地方）：

1. **分类：新增功能 vs 体验优化** —— 套用 skill 的「判断表」。**产出物是全新能力/独立模块/独立页面/独立文档位 → 新增功能**，哪怕它挂在已有页面上、动机是"提升体验"也算新增功能。反例教训：`创建人员群组`、`项目测试文档` 是全新能力，应归**新增功能**，我却因"挂在已有页面"错判成体验优化。
2. **影响强度** —— 套用 skill 的「强度规则表」。新增用户可感知的核心能力 → 至少**中等**，不要保守判"较小"。还要核对本次发版实际范围是否超出验收清单（可能含未验收的模块，如 V2.2.4 还有"模型外包""我的任务日历"），强度按发版全集判。统一用 `较小/中等/较大`。
3. **BUG 转正向表述** —— 对外发版**不暴露"曾出数据异常/误建"等负面信息**。把"修复了XX异常"改写成正向能力，如"仅支持在子需求下创建任务，规范需求选择范围，防止错选"。内部 BUG（尤其接口层发现）放到报告的「交付前需人工确认」节，不进对外发版正文。
4. **价值括号写业务结果，不写功能能力** —— 套用 skill 的「价值描述红黑榜」。写"降低XX创建成本/减少人工汇总/提高盘点效率"，不写"方便统一录入/支持XX"（能接在"支持"后的多半是功能能力，不合格）。

写完后**用 skill 的「常见错误」表自查一遍**再交付。发版内容以用户人工定稿为最终版；被纠正后把差异作为反例回填本节。

### 第 6 步：交接发版技能（可选，问用户）

- 报告与截图目录路径发回用户。**路径先跑 `wslpath -w` 确认真实 Windows 路径再发，不要凭目录名猜。**
- 定妆图不需要另行拷贝：发版内容直接引用 `acceptance/{版本}/final-*.jpg` 原路径（2026-07-09 已取消「发版最新资源截图」暂存目录，验收产物和发版素材同源）。

## 安全红线

1. 验收会产生写操作（建任务、改状态等）——**测试环境**允许直接做，测试数据**不需要清理、不需要在报告里说明**（2026-07-09 用户确认：本来就是测试环境）；**正式环境**每个写操作前单独问用户。
2. 不删除任何已有数据，包括测试环境。
3. 验收中遇到登出、企微扫码、二次验证：暂停，问用户。
4. 两套环境 host 不外发（不写进给外部看的文档/截图标注）。
